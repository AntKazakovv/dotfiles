# Система релизов

## Дисклеймер

**Важно 1**: Для корректной работы на локальном АРМ должны быть установлены jq и python3

**Важно 2**: Для корректной системы формирования change logs в корне wlc-engine должен лежать файл с ключем api доступа <https://wlcgitlab.egamings.com/wlc/devtools/-/blob/master/theme/change-logs.md>

## Процедура создания релиза

1. В процессе подготовки релиз кандидата (RC) движка тикеты агрегируются с меткой **awaiting-engine-release**, MRы из тикетов последовательно принимаются и тикеты переводятся в статус Released

2. После приема всех необходимых МРов для создания RC версии необходимо выполнить команду

    ```bash
    ./make_release.sh rc
    ```

    Будет автоматически сформирован change logs и создан тег следующий от текущего с суффиксом rc.X, где X версия RC начиная с 1

    ! **Важно** с момента создания RC никакие изменения, кроме фиксов самого RC в master не принимаются

3. После успешной сборки необходимо обновить проекты <https://wiki.egamings.com/pages/viewpage.action?pageId=92212241>, для чего выполнить команду

    ```bash
    ./update_devcasino.sh
    ```

    В процессе выполнения скрипта будут выводиться запросы на подтверждение продолжения с выводом _git status_ проекта.
    До подтверждения можно вручную вносить изменения в обновляемый проект во временной папке ./temp/repo от корня движка
    Также у скрипта есть параметр _no-docker_, который позволяет переключить использование докера для обновления **npm** зависимостей.
    Зависимости композера обновляются только через вызов соответсвующего контейнера докера.

    ! **Важно**: ветки devcasino и прочих проектов не должны конфликтовать с develop

4. После успешной сборки всех необходимых проектов и веток создается тикет на тестирование релиза с заголовком "Тестирование релиза X.Y.Z", где в качестве X.Y.Z следующая версия движка.

   В тикет релейтами добавляются все тикеты, попавшие в релиз и ссылка на change log релиз кандидата, после чего тикет переводится на группу _I_Fronttest_.

5. В случае если тикет на тестирование вернулся с _Failed_

   * тикеты, по которым выявлены проблемы возвращаются с статусом _Failed_ на ответственных разработчиков, разработчики дополнительно тегаются в Slack.

        **Важно**: фиксы RC выполняются в приоритетном порядке (исключая наличие SLA)

   * после того как все необходимые фиксы выполнены и изменения приняты в master создается следующий RC аналогично пп. 2-3

6. После успешного завершения тестирования релиз кандидата (тикет вернулся с _Verified_) создается стабильный релиз выполнением команды:

    ```bash
    ./make_release.sh release
    ```

    и производится обновление веток:

    ```bash
    ./update_devcasino.sh
    ```

7. После этого тикеты с меткой **awaiting-engine-release** переводятся дальше по назначению (на авторов либо на обновление проектов, если это необходимо) с пометкой _Принято, релизнуто_  удалением метки **awaiting-engine-release**

8. С этого момента можно начинать обрабатываться тикеты и принимать MRы в следующий релиз

## Работа с тикетами с момента создания RC движка

Тикеты стандартно проходят процедуру тестирования и код ревью, после прохождения всех этапов помечаются меткой _await X.Y.Z_, где X.Y.Z - ожидаемая стабильная версия релиза. Статус тикетов _Verified_.

## Создание хотфикс версии движка

1. В случае когда на проекте нужно внести исправления в движок создается хотфикс релиз от версии движка в мастере проекта, для этого сначала создается hotfix ветка выполнением команды:

    ```bash
    ./make-hotfix-branch.sh X.Y.Z ticketNumber
    ```

    где

    * X.Y.Z - версия движка в мастере проекта

    * ticketNumber - номер тикета по которому происходит внесение изменений

    например:

    ```bash
    ./make-hotfix-branch.sh 1.3.69 101010
    ```

после выполнения команды будет создана ветка **hotfix-[YYYYMMDD]-[ticketNumber]**

1. Далее необходимо убедиться что последний коммит в ветке соответствует коммиту необходимого релиза, обычно это коммиты вида _Updated for release "@egamings/wlc-engine" X.Y.Z_

2. Внести и запушить необходимые изменения

3. создать хотфикс тег выполнив команду

    ```bash
    ./make_release.sh release
    ```

    Будет создан тег _X.Y.Z.n_

4. После успешной сборки тега в дженкинсе необходимо залочить версию движка в master проекта на версии _X.Y.Z.n_ и выполнить обновление зависимостей командой:

    ```bash
    rm -rf node_modules  package-lock.json && npm install
    ```

5. После чего, проверив корректность применения изменений, вывести их стандартным способом

6. После, для внесения изменений в движок, если это необходимо, создается тикет, который решается в рабочее время в приоритетном порядке.

! **Важно**: Дальнейшее обновления проекта возможны только если внесенные хотфиксом изменения попали в стабильный релиз движка, иначе все последующие необходимые хотфиксы выводятся по этой же схеме

## Дополнительные возможности релизного скрипта (make_release.sh)

1. Запуск с параметром _minor_ поднимает минорную версию Y в наборе X.Y.Z, Z при это обнуляется

    для примера при запуске с текущей версии _1.3.70_

    ```bash
    ./make_release.sh minor
    ```

    будет сформирован тег _1.4.0_

    **Важно** выполняется только из ветки master

2. Запуск с параметром _major_ поднимает мажорную версию X в наборе X.Y.Z, Y и Z при это обнуляются

    для примера при запуске с текущей версии _1.3.70_

    ```bash
    ./make_release.sh major
    ```

    будет сформирован тег _2.0.0_

    **Важно** выполняется только из ветки master

3. В качестве параметра также можно указать конкретную версию в формате X.Y.Z, тег которой и будет сформирован

    для примера при запуске с текущей версии _1.3.333_

    ```bash
    ./make_release.sh 1.3.333
    ```

    будет сформирован тег _1.3.333_

    **Важно** выполняется только из ветки master

4. Эта и дополнительная информация достпны в инлайн документации скрипта при его запуске без параметров:

    ```bash
    ./make_release.sh
    ```

## Добавление проектов и веток в скрипт автообновления

Скрипт **update_devcasino.sh** позволяет добавить проекты и ветки к системе автообновлений, для этого нужно добавить строчки в файл

```bash
projects[X]="";
branches[X]="";
```

Здесь X - это следующий порядковый номер записи в массиве проектов.
В переменную projects[X] заносится ssh адрес репозитория.
В переменную branches[X] заносятся ветки, которые необходимо обновлять или создавать

* develop  - обновление QA среды
* test     - создание тестового тега
* master   - merge develop в master и создание прод тега
* scrXXXX  - обновление (rebase) ветки
* +scrYYYY - создание ветки и push с форсом

для примера

```bash
branches[X]="develop test master scr0 +scr1"
```

* обновит зависимости в develop
* создать test тег
* сделает merge и создаст прод тег
* сделает rebase ветки scrXXXX с девелопом
* создаст копию ветки develop с именем scrYYYY и пушнет ее с форсом
