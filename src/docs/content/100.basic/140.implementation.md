# 1.4. Implementation features

## WLC Engine - is a service

WLC Engine is the basic service for the deployment of client WLC projects. This is why its implementation allows for making changes to the functionality and visual appearance of new projects without affecting the already existing WLC Engine projects.

In addition, WLC Engine allows for regular updates to its functionality, such as bug fixing, new skins for components, new modules and components, and new functions for the existing modules and components.

The WLC Engine is a website builder without a fixed mesh of components. All components can be set up using configuration files, including the layout, content, behavior and appearance of each component and page. In addition, the system offers ready-to-use and thoroughly tested default configuration files right out of the box to ensure that all parts of your WLC project will work fine.

## Styles

The CSS styles are implemented as ready-to-use standard themes that can be flexibly switched and adjusted using SASS variables for each component. In other words, each component features at least one standard theme that was thoroughly configured and tested by our QA specialists. All basic and optional numeric values are set up using SASS variables, which can be easily edited at the project level without rewriting any code of your component's CSS. This implementation makes it possible to install system-wide updates without affecting project-wide custom designs.

For example, let's assume that we have a game catalog component with a standard layout, where the names of games are listed in 5 columns and 6 rows, with 10px rounding of blocks, a white title, 20px font size, and 20px spacing between blocks. And now we have to modify all these values. The typical solution is to include your CSS stylesheet in the project and then define the appropriate styles for these elements. However, if after code refactoring in the next release of the WLC Engine the class names or the HTML template structure are changed (e.g. to fix any component display bugs in Safari), the design styles may become invalid and lead to unpredictable results. But if we modify only variables, they will remain linked to specific template elements after any update and the problem won't even appear.

When you need to make major style changes that cannot be handled using only variables, feel free to define your own styles below in the code as usual and override default parts of the CSS code for your component. And even this is not enough, you can disable all standard styles for the component and write your own ones.

## Component templates

Each component features its own configured HTML template. If it doesn't fit the intended functionality, you can assign a custom template to the component in the project without making any changes to the WLC Engine files. This allows you to get TS/JS updates for the template code base without any dependencies on the HTML/CSS code.

# 1.4. Особенности реализации

## WLC Engine - сервис

WLC Engine представляет собой сервис, на основе которого разравачиваются клиентские WLC проекты. В связи с этим его
реализация выполнена таким образом, чтобы была возможность вносить изменения в функционал и внешний вид проекта так,
чтобы не затронуть другие проекты, построенные на WLC Engine.

Кроме того WLC Engine подразумевает постоянное обновление функционала: багфиксинг, добавление новых тем оформления
компонентов, добавление новых модулей и компонентов, расширение функционала существующих модулей и компонентов.

WLC Engine представляет собой конструктор сайта, не имеющий строгой сетки компонентов. Всё задаётся через файлы
конфигурации: положение компонентов на странице, состав каждой страницы, её поведение, внешний вид. При этом "из
коробки" предоставляются готовые файлы конфигурации, полностью настроенные, протестированные и гарантирующие корректную
работу всех частей WLC проекта.

## Стилизация

В основу стилизации (CSS) заложен принцип смены стандартных тем оформления компонентов с внесением в них изменений на
уровне SASS переменных. Т.е. каждый компонент имеет как минимум одну стандартную тему оформления, полностью настроенную
и протестированную отделом QA. Все основные числовые значения (а также другие возможные) вынесены в SASS переменные,
которые можно поменять на уровне проекта, не переписывая CSS компонента. Таким образом сохраняется возможность получения
централизованных обновлений с сохранением кастомного оформления.

Пример: допустим компонент каталога игр имеет стандартное оформление, в котором игры выводятся в 5 столбцов и 6 строк,
имеют скругления блоков 10px, заголовок белого цвета, размер шрифта 20px, расстояние между блоками 20px. Необходимо
сменить эти значения на иные. В классическом варианте можно подключить свой файл стилей CSS в проект и определить
соответствующие стили для данных элементов. Если после рефакторинга кода в новом релизе WLC Engine сменится названия
классов или структура HTML шаблона (как вариант это может понадобиться для фикса багов отображения компонента в браузере
Safari) это может сделать проектные стили недействительными и привести к непредсказуемому результату. В случае изменения
переменных не теряется их привязка к конкретным элементам шаблона и любое обновление произойдёт гладко.

В случае, когда стилизовать компонент одними переменными недостаточно можно пойти классическим путём и определить свои
стили ниже по коду, переопределяя те части CSS кода, что заданы для компонента по умолчанию. Если и этого недостаточно,
есть возможность отключить у компонента все стандартные стили и написать свои.

## Шаблоны компонентов

Каждый компонент имеет свой настроенный HTML шаблон. В случае, если он не подходит для реализации задуманного
функционала, есть возможность назначить компоненту свой шаблон в рамках проекта, не внося изменения в файлы WLC Engine.
Это позволяет получать обновления кодовой базы шаблона в части TS/JS, при этом не зависеть от HTML/CSS кода.
