# 5.1. Standard component

## Example of a simple component

```typescript
import {
    Component,
    Inject,
    OnInit,
    Input,
    ChangeDetectionStrategy,
} from '@angular/core';

import {AbstractComponent} from 'wlc-engine/classes/abstract.component';
import {ConfigService} from 'wlc-engine/modules/core';

import * as Params from './scratch.params';

import {
    get as _get,
} from 'lodash';

export {IScratchCParams} from './scratch.params';

@Component({
    selector: '[wlc-scratch]',
    templateUrl: './scratch.component.html',
    styleUrls: ['./styles/scratch.component.scss'],
    changeDetection: ChangeDetectionStrategy.OnPush,
})
export class ScratchComponent extends AbstractComponent implements OnInit {
    public $params: Params.IScratchCParams;

    @Input() protected inlineParams: string;

    constructor(
        @Inject('injectParams') protected injectParams: Params.IScratchCParams,
        protected configService: ConfigService,
    ) {
        super({injectParams, defaultParams: Params.defaultParams}, configService);
    }

    public ngOnInit(): void {
        super.ngOnInit(this.inlineParams);
    }
}
```

## Requirements to the component's program code

### The order of the "import" statements must be as follows:

* External libraries, such as Angular, RxJS, etc.
* WLC Engine's external dependencies, such as services, helpers, constants, etc.
* Internal parameters and interfaces related to a given component.
* Methods from the Lodash library.
* Export of the interface for component parameters.

### Implementation features of the @Component decorator's parameters

* The selector is specified as attribute (in square brackets).
* The default strategy is OnPush.
* The component template must be an external HTML file.

### Implementation features of the component classes

* Declaring the $params parameter with the type of the component parameters. It is required for the proper operation of IDEs and for processing it as typed object. When it is not declared, the component will still be able to work, since this parameter is declared in the component's abstract class, but won't have a data type.
* Import inlineParams only when they are really needed, e.g. when you want to retrieve parameters from the parent component to a child component.
* injectParams is a mandatory parameter for the constructor.
* ConfigService is an optional parameter. It is required only when the component is being developed with global parameters by default.
* super() is the mandatory transfer of the {injectParams, defaultParams} object.
* super.ngOnInit(): make sure to call this method when there is an ngOnInit method in the class. If the component uses inlineParams as input parameter, always pass it as parameter: super.ngOnInit (this.inlineParams).

### General implementation features

* By default, the interfaces and parameters are imported using the Params alias.

## How to use the abstract component

The abstract component implements the key workflows, such as processing of modifiers, passing of parameters and creating of a base class.

### Component parameters

All components which are based on the abstract component can automatically generate an object of parameters based on the data from the following three sources:

1. **Standard parameters** imported into the component class at the code creation stage.
2. **Injectable parameters** received by the component as single injectable object.
3. **Inline parameters** obtained via the @Input() decorator directly within the component.

Parameters are indicated with respect to their weights when the final `this.$params` object is being generated. Thus, inline parameters take precedence over others.

### Base class

n the BEM methodology, it is a block class. This class is responsible for all styles applied to component templates. If you change the styles, then all the default styles will no longer be used.  

Make sure to set a unique base class for each component, regardless of the encapsulation of the styles provided by the Angular framework itself.  

You can set or modify this using the standard parameters.
Since this parameter is available as `public $class: string` via the abstract class, it is used as `{{$class}}` in the HTML template.

### Modifiers

All changes to the component's state must be reflected in modifiers. This will allow you to flexibly manage their styling without losing the ability to customize them.  

Let's suppose that you have a button component. CSS styles update the base class of this component only with the basic parameters, such as resetting sizes, colors, or other parameters specific for all buttons. Then, the necessary styles responsible for the component's appearance are added for the default theme modifier (**theme-default**).  

Though type modifiers are usually responsible for the functional behavior of a component, they can also participate in its styling. For example, `'click'` and `'hover'` modifiers of the **LanguageSelector** components are also used in the CSS styles.

### How to create a new component

1. Import the abstract component and then assign it as the parent class for the new component.
2. In the constructor's parameter, declare `@Inject('injectParams') protected params: unknown` (specify the type). The injector will update this parameter with the object you've specified in the component's parameters when describing the page configuration in **LayoutsConfig**.
3. Call the constructor of the parent class in the component constructor: `super(<IMixedParams>{injectParams, defaultParams});` to generate the `this.$params` object.
4. When your components contains the `ngOnInit` method, make sure to call the parent method `super.ngOnInit()` within the component and pass the optional `inlineParams` object.

### Implicit methods/parameters in the abstract class

1. **`ngOnDestroy()`** implements the completion of the `this.$destroy` thread. This thread is mainly used for automatic unsubscribing from events by means of the `pipe(takeUntill(this.$destroy))` filter. If this method (** ngOnDestroy) is present in your component, make sure to call the `**super.ngOnDestroy();` method.
2. **`ngOnInit()`** automatically mixes inline parameters, such as type, theme, themeMod and customMod, to the `this.$params` object. Other parameters must be passed to the `super.ngOnInit()` method within the class of your component.

## How to work with modifiers

When the component starts up, the following modifiers are added to the parent container by default:

* **{{$class}}--theme-default**
* **{{$class}}--theme-mod-default**
* **{{$class}}--type-default**

### You should describe the styling of your component based on these modifiers.

* Make sure to develop styles only within modifiers.
* If you want to use another theme, then develop new styles. It is recommended that you place all styles that overlap between themes in mix-ins.
* Theme modifier styles must also be linked to the theme and shouldn't be applied without it.
* Type modifiers are not required to be bound to a theme in most cases.

# 5.1. Стандартный компонент

## Пример простого компонента

```typescript
import {
    Component,
    Inject,
    OnInit,
    Input,
    ChangeDetectionStrategy,
} from '@angular/core';

import {AbstractComponent} from 'wlc-engine/classes/abstract.component';
import {ConfigService} from 'wlc-engine/modules/core';

import * as Params from './scratch.params';

import {
    get as _get,
} from 'lodash';

export {IScratchCParams} from './scratch.params';

@Component({
    selector: '[wlc-scratch]',
    templateUrl: './scratch.component.html',
    styleUrls: ['./styles/scratch.component.scss'],
    changeDetection: ChangeDetectionStrategy.OnPush,
})
export class ScratchComponent extends AbstractComponent implements OnInit {
    public $params: Params.IScratchCParams;

    @Input() protected inlineParams: string;

    constructor(
        @Inject('injectParams') protected injectParams: Params.IScratchCParams,
        protected configService: ConfigService,
    ) {
        super({injectParams, defaultParams: Params.defaultParams}, configService);
    }

    public ngOnInit(): void {
        super.ngOnInit(this.inlineParams);
    }
}
```

## Требования к коду компонента

### Порядок следования инструкций import:

* Сторонние библиотеки: Angular, RxJS и прочие.
* Внешние зависимости WLC Engine - сервисы, хелперы, константы и прочие.
* Внутренние параметры и интерфейсы, относящиеся к данному компоненту.
* Методы библиотеки Lodash.
* Экспорт интерфейса параметров компонента.

### Особенности параметров декоратора @Component

* Селектор указывается как атрибут (в квадратных скобках).
* Применяется по умолчанию стратегия OnPush.
* Шаблон компонента - обязательно внешний HTML файл.

### Особенность класса компонента

* Объявление параметра $params с типом параметров компонента. Необходим для корректной работы IDE а так же для работы с
  ним как с типизированным объектом. Если не объявить, то работать компонент будет, так как данный параметр объявлен в
  абстрактном класса компонента, но ему не присвоен тип данных.
* Импорт inlineParams по необходимости, когда требуется "прокинуть" параметры из родительского компонента в дочерний.
* injectParams - обязательный параметр для конструктора.
* ConfigService - необязательный параметр. Требуется в случае, если компонент разрабатывается с глобальными параметрами
  по умолчанию.
* super() - обязательная передача объекта {injectParams, defaultParams}.
* super.ngOnInit() - обязательно вызывать метод при наличии в классе метода ngOnInit. В случае, если у компонента есть
  входящий параметр inlineParams - обязательная передача его в качестве параметра: super.ngOnInit(this.inlineParams).

### Общие особенности

* Импорт интерфейсов и параметров по умолчанию производится через алиас Params.

## Принцип работы с абстрактным компонентом

Абстрактный компонент отвечает за следующие основные принципы работы: работа с модификаторами, передача параметров,
формирование базового класса.

### Параметры компонента

Все компоненты, основанные на абстрактом компоненте, получают возможность автоматического формирования объекта
параметров на основе данных из 3-х источников:

1. **Стандартные параметры**, импортируемые в класс компонента на стадии созданий кода.
2. **Инжектируемые параметры**, получаемые компонентом как один инжектируемый объект.
3. **Инлайн параметры**, получаемые через декоратор @Input() непосредственно в самом компоненте.

Параметры указаны в соответствии с их весом при формировании итогового объекта this.$params. Таким образом инлайн
параметры имеют высший приоритет перед остальными.

### Базовый класс

В методологии BEM - класс блока. Отвечает за все стили, применяемые к шаблоны компонента. В случае смены все стили,
используемые по умолчанию, более не используются.

Для каждого компонента в обязательном порядке необходимо задавать уникальный базовый класс, невзирая на инкапсуляцию
стилей самого Angular фреймворка.

Задаётся через стандартные параметры, при необходимости переназначается.

Через абстрактный класс данный параметр доступен как **public $class: string**, таким образом в HTML шаблоне применяется
как
**{{$class}}**.

### Модификаторы

Все изменения состояния компонента должны отражаться в модификаторах. Это позволит гибко управлять их стилизацией, при
этом сохранив возможность внесения индивидуальных изменений.

Допустим у нас есть компонент кнопки. На базовый класс через CSS стили к нему добавлены только основные параметры: сброс
размеров, цвета, другие параметры, одинаковые для всех кнопок вообще. Далее для модификатора темы по умолчанию (
**theme-default**) добавляются необходимые стили, отвечающие за оформление компонента.

Модификаторы типа отвечают в большинстве случаев за функциональное поведение компонента, но так же могут участвовать и в
стилевом оформлении. Например **'click'** и **'hover'** модификатор компонента **LanguageSelector** так же участвуют в
CSS стилях.

### Порядок создание нового компонента

1. Импортировать абстрактный компонент, назначить его как родительский класс для нового компонента.
2. В параметрах конструктора объявить **@Inject('injectParams') protected params: unknown** (указать тип). При этом в
   данный параметр инжектор передаст тот объект, что указан в параметрах компонента при описании конфигурации страницы в
   **LayoutsConfig**.
3. Вызвать в конструкторе компонента конструктор родительского класса: **super(<IMixedParams<unknown>>{injectParams,
   defaultParams});** Это сформирует объект **this.$params**
4. Если в компоненте есть метод **ngOnInit** - вызвать в нём метод родителя **super.ngOnInit()** и передать в него (если
   требуется) объект **inlineParams**.

### Какие неявные методы/параметры есть в абстрактном классе

1. **ngOnDestroy()** - реализует завершение потока **this.$destroy**. Данный поток используется в основном для
   автоматической отмены подписки на события через фильтр **pipe(takeUntill(this.$destroy))**. Если данный метод (**
   ngOnDestroy**) присутствует в компоненте, необходимо не забыть вызвать метод **super.ngOnDestroy();**
2. **ngOnInit()** - автоматически подмешивает инлайн параметры **type**, **theme**, **themeMod**, **customMod** в
   объект **this.$params**. Прочие параметры необходимо передавать в метод **super.ngOnInit()** в классе самого
   компонента.

## Принцип работы с модификаторами

При старте компонента по умолчанию к родительскому контейнеру добавляются следующие модификаторы:

* **{{$class}}--theme-default**
* **{{$class}}--theme-mod-default**
* **{{$class}}--type-default**

### Необходимо в рамках данных модификаторов описать стилевое оформление компонента.

* Стили разрабатываются только внутри модификаторов.
* Для другой темы - разрабатываются новые стили. Все пересечения стилей между темами лучше всего вынести в миксины.
* Стили модификаторов темы так же должны быть привязаны к теме, т.е. без темы они не должны применяться.
* Модификаторы типа могут не быть привязаны к тебе, если это возможно.
